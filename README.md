# 项目代码量与工作量评估工具

> 一套适用于 Java / Web / 移动端项目的代码规模统计与人日预估工具，支持前后端独立拆分估算。

---

## 🚀 快速运行

本工具提供三个版本，功能完全一致，按您的环境选择其一即可：

### 方案 A — Shell 脚本（macOS / Linux 推荐）

```bash
# 复制脚本到您的项目根目录，然后运行：
./Project\ Estimator.sh
# 或
./estimate.sh
```

### 方案 B — Java（跨平台，Windows / macOS / Linux 通用）

```bash
# 1. 将 ProjectEstimator.java 复制到项目根目录
# 2. 编译
javac ProjectEstimator.java
# 3. 运行
java ProjectEstimator
```

### 方案 C — Node.js

```bash
# 复制 estimate.js 到项目根目录，然后运行：
node estimate.js
```

> **重要**：请在项目的**根目录**下运行，工具会自动递归扫描子目录。

## ⚙️ 可调整的参数

所有参数均集中在每个脚本的顶部，方便修改。

### 1. 生产力基准（最常用）

| 参数 | 默认值 | 含义 |
|------|--------|------|
| `BE_LOW_BOUND` | `80` | 后端最高效率（行/人日），区间下限 |
| `BE_HIGH_BOUND` | `120` | 后端最低效率（行/人日），区间上限 |
| `BE_RECOMMENDED` | `100` | 后端推荐基准，用于计算"推荐参考值" |
| `FE_LOW_BOUND` | `150` | 前端最高效率（行/人日） |
| `FE_HIGH_BOUND` | `200` | 前端最低效率（行/人日） |
| `FE_RECOMMENDED` | `175` | 前端推荐基准 |
| `COMPLEXITY_FACTOR` | `1.2` | 复杂度系数（含需求沟通、测试等额外开销） |

> **如何理解区间**：`工作量区间 = (加权代码行数 / 效率) × 复杂度系数`
> - 区间**下限** = 用 `HIGH_BOUND`（高效模式）计算
> - 区间**上限** = 用 `LOW_BOUND`（保守模式）计算

**调整建议**：
- 团队水平高、需求稳定 → 适当**提高** `LOW_BOUND`（缩小上限）
- 项目复杂、需求多变 → 适当**提高** `COMPLEXITY_FACTOR`（如调至 `1.5`）

### 2. 文件权重

| 权重 | 适用文件类型 | 说明 |
|------|------------|------|
| `1.0` | `.java` `.tsx` `.ts` `.js` `.sql` `.jsp` | 核心逻辑，含金量高 |
| `0.5` | `.xml` `.yml` `.css` `.html` `.properties` | 配置/样式，相对机械 |

### 3. 排除目录

以下目录会被**自动跳过**，无需手动设置：

`node_modules` / `.next` / `.git` / `target` / `bin` / `build` / `dist` / `classes` / `.gradle` / `.idea`

如需添加自定义排除目录，在脚本中搜索 `EXCLUDE_DIRS` 修改即可。

## ⚠️ 使用注意事项

> **前期框架搭建阶段**：项目初始化时通常会有大量脚手架代码（如 Spring 配置、生成的 stub 文件），LOC 容易虚高。建议此时以**估算下限**作为参考基准，待功能迭代稳定后再重新评估。

> **统计结果为 0**：请确认是否在含有源码的目录下运行，并检查输出的"文件发现摘要"，确认哪些类型的文件被识别到了。
